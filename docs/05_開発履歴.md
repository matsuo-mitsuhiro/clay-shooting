### v3.74 — 参加人数集計の修正 & 閲覧履歴機能の削除（2026-04-19）

   - 従来: `members` テーブルから `COUNT(DISTINCT member_code)` GROUP BY class

     - 1選手=1行、両日参加者も1名としてカウント
     - 申込管理リストに表示されている人数と一致
   - 対象: `app/api/tournaments/[id]/report/route.ts`、`app/api/tournaments/[id]/report-excel/route.ts`
   - AAA クラスも集計対象に追加（migration 019 対応漏れ）、ReportTab UI にも AAA 行追加
2. **閲覧履歴機能を完全削除**:
   - 管理画面の「閲覧履歴」タブを削除
   - 閲覧者ログイン API (`/api/tournaments/[id]/viewer-login`) からログ記録部分を削除（ログインによる行ハイライト機能は残す）
   - 閲覧ログ API (`/api/tournaments/[id]/viewer-logs`) を削除
   - `components/admin/ViewerHistoryTab.tsx` を削除
   - `lib/types.ts` から `ViewerLog` 型を削除
   - migration 023 で `viewer_logs` テーブルを DROP
3. **変更ファイル**:
   - `app/api/tournaments/[id]/report/route.ts`、`app/api/tournaments/[id]/report-excel/route.ts`
   - `components/admin/ReportTab.tsx`、`components/admin/AdminDetail.tsx`
   - `app/api/tournaments/[id]/viewer-login/route.ts`（簡素化）

### v3.73 — 会員番号の編集も申込管理に集約（2026-04-19）
1. **選手管理タブの編集を「所属」「賞典外」のみに縮小**:
   - 会員番号 input を廃止 → **表示のみ**
   - クラス: disabled select を廃止 → **バッジ表示のみ**（編集モードでも同じ）
   - 審判: disabled checkbox を廃止 → **⚑アイコン表示のみ**（編集モードでも同じ）
   - 理由: disabled UI でも「権限があれば編集できる」と誤解を招くため、UI要素自体を廃止
2. **申込管理タブで会員番号を編集可能に**:
   - 会員番号列に input を追加（編集モード時）
   - バリデーション: 半角数字のみ（全角→半角自動変換）、空不可
   - 重複チェック: 同一大会内の別アクティブ申込と衝突しないか
   - **未登録警告モーダル**: 選手マスター (player_master) に存在しない会員番号を入力した場合は確認モーダル表示、「このまま保存」で続行可
3. **API 同期（申込管理で会員番号変更時）**:
   - `registrations.member_code` → 新コード
   - `members.member_code`（両日分）→ 新コード（移行済みの場合）
   - `scores.member_code` → 新コード（紐付け維持）
   - `player_master` は触らない（主キーなので rename 不可、新コードは既存選手を指すか未登録として解釈）
4. **選手管理 API の防御強化**:
   - `PATCH /api/tournaments/[id]/members/[memberId]` は `body.member_code` / `class` / `is_judge` を無視
   - 選手管理経由では `belong` / `is_non_prize` のみ変更可能
5. **変更ファイル**: `components/admin/MembersTab.tsx`（大幅縮小）、`components/admin/RegistrationsTab.tsx`、`app/api/tournaments/[id]/members/[memberId]/route.ts`、`app/api/tournaments/[id]/registrations/[regId]/route.ts`、`lib/version.ts`

### 検討した代案・却下理由
- **案: disabledグレーアウト（v3.72と同方針）** → 「権限があれば編集できる」と誤解を招くため却下
- **案: 会員番号変更時に player_master も rename** → 主キー変更は他大会との整合性を壊すリスク大、却下

### v3.72 — クラス・審判の編集を申込管理に集約（2026-04-19）
1. **選手管理でクラス・審判を編集不可に**:
   - `MembersTab` のクラスドロップダウン・審判チェックボックスを `disabled`（グレーアウト）に
   - ホバー時に「クラスの変更は『申込管理』タブから行ってください」とツールチップ表示
   - 編集モードでもマウスオーバー時に `cursor: not-allowed` 表示
2. **members API の防御的対応**:
   - `PATCH /api/tournaments/[id]/members/[memberId]` で body の `class` / `is_judge` を無視
   - 選手管理経由で `player_master` が更新されることはなくなった（申込管理に一本化）
3. **申込管理 → player_master 同期バグ修正**:
   - 従来: `UPDATE player_master SET class = ?` ← `class` カラムは migration 019 で `trap_class` / `skeet_class` に分離済みのため**サイレント失敗**していた
   - 修正: 大会の `event_type` に応じて `trap_class` / `skeet_class` を振り分けて更新
   - これにより申込管理でクラスを編集 → registrations + members(両日) + player_master の3箇所が一括更新される
4. **同期動作まとめ（申込管理でクラス/審判を変更したとき）**:
   - `registrations` テーブル: ✓
   - `members` テーブル（両日分）: ✓（移行済みの場合）
   - `player_master`: ✓（今回バグ修正）
5. **変更ファイル**: `components/admin/MembersTab.tsx`、`app/api/tournaments/[id]/members/[memberId]/route.ts`、`app/api/tournaments/[id]/registrations/[regId]/route.ts`、`lib/version.ts`

### 背景
- 大会9 で `member_code` 35345/35346/38161 が `day1=B, day2=C` になっていた
- 原因: 選手管理でクラスを変更した際、当該日の `members` 行のみ更新で両日同期していなかった
- さらに申込管理経由でも `player_master` の更新先カラム名が誤っておりサイレント失敗していた
- 変更の入り口を「申込管理タブ」一本に集約し、そこから全テーブルを同期する方式に整理

### v3.71 — 賞典外を一覧に統合 & 2日目のみ選手を表示（2026-04-19）
1. **v_results ビューを修正（migration 022）**:
   - `day1_members LEFT JOIN day2_members` → `FULL OUTER JOIN` に変更
   - **不具合修正**: 2日目のみ登録の選手が成績一覧に出てこなかった問題を解消
   - 氏名・所属・クラス・審判・賞典外フラグは `COALESCE(day1, day2)` で補完
   - 賞典外選手は `is_non_prize = TRUE` の条件で RANK() 対象から除外 → `rank = NULL` で返す
2. **成績一覧の賞典外表示方式を変更**:
   - 従来: 「── 賞典外 ──」区切り線で別ブロック化
   - 新仕様: **一覧に混在**（合計降順）、氏名横に「**（賞典外）**」と小さく表示
   - 賞典外選手は順位番号なし、通常選手は連番で採番（賞典外は順位計算に影響しない）
   - 「組」列は 1日目のみ選手は `N組` / 両日は `N/M組` / 2日目のみ選手は `M組` 形式
3. **大会記録審査表Excelに全員掲載**:
   - 賞典外選手を含めて出力（従来は `rank NULLS LAST` で末尾だったが、点数順で混在に）
   - 氏名列に「氏名（賞典外）」として追加表示
   - 最大72名の制限は維持
4. **手動順位設定モーダル**: DnDリストから賞典外選手を除外（賞典外は手動順位対象外）
5. **ORDER BYの統一**: 成績API・記録審査表Excel ともに
   `(失格・棄権を末尾) → total DESC → カウントバック → CB ASC → FR DESC`
6. **変更ファイル**: `db/migrations/022_non_prize_inline_ranking.sql`、`app/api/tournaments/[id]/results/route.ts`、`app/api/tournaments/[id]/inspection-report/route.ts`、`components/admin/ResultsTab.tsx`、`components/viewer/ViewerPage.tsx`、`lib/version.ts`
7. **⚠️ migration 022 はNeonで実行済み**

### 検討した代案・却下理由
- **案A: 賞典外を1日目/2日目の2ブロックに分割表示** → 複雑でメンテ負担大、却下
- **案B: 合計スコアで高い方の日のブロックに配置** → 両日選手の扱いが曖昧、却下
- **採用: 賞典外を一覧に混在＋「（賞典外）」ラベル** → シンプルで直感的

## 却下・保留になった案

| 案 | 内容 | 理由 |
|----|------|------|
| 選手削除 A案 | 入力フォームのみクリア | 機能不足 |
| 選手削除 B案 | すべて確認ダイアログ付き削除 | 煩雑 |
| 選手削除 C案 | 入力クリア+DB削除（点数ありのみ確認） | **採用** |
| 空きスロット A案 | 薄いオーバーレイのみ | 見づらい |
| 空きスロット B案 | ゴールド左ボーダー+「－（空き）」テキスト | **採用** |

## 既知の問題・制約

| 問題 | 状況 | 対応 |
|------|------|------|
| WindowsローカルでEPERMエラー | `next build` 失敗 | Vercelでビルド（Linux）は正常 |
| PostgreSQL数値型→文字列返却 | NUMERICはJSでstring | `Number(x).toFixed()` で変換 |
| 会員番号なしの選手は点数登録不可 | scoresのmember_codeはNOT NULL | 仕様通り、UIで警告表示 |
