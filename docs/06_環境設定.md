## Google OAuth セットアップ

### Google Cloud Console 設定手順
1. https://console.cloud.google.com/ にアクセス
2. プロジェクト選択 → 「APIとサービス」→「認証情報」
3. 「OAuthクライアントIDを作成」→ アプリケーションの種類: **ウェブアプリケーション**
4. 承認済みのリダイレクトURI に追加:
   - **本番**: `https://clay-shooting.vercel.app/api/auth/callback/google`
   - **ローカル**: `http://localhost:3000/api/auth/callback/google`
5. クライアントID・クライアントシークレットを取得 → Vercel と `.env.local` に設定

### 許可メールの管理
- `SYSTEM_ADMIN_EMAILS` にカンマ区切りで設定（例: `a@x.com,b@y.com`）
- ここに無いメールでGoogleログインしても認証拒否

## デプロイ手順（3環境フロー、v3.78.5〜）

### 開発から本番リリースまでの流れ
```bash
# 1. feature ブランチで作業
git checkout -b feature/<目的>

# 2. 変更 + 各種チェック
npx tsc --noEmit --incremental false   # TypeScript
npx eslint .                            # ESLint（エラー0件必須）

# 3. lib/version.ts と docs/05・07 更新
# 4. コミット
git add <ファイル>
git commit -m "..."

# 5. push + Staging向けPR作成
git push -u origin feature/<目的>
gh pr create --base staging --title "..." --body "..."

# 6. CI（GitHub Actions）が tsc/lint/build を自動実行
# 7. ユーザーが PR をレビュー → staging にマージ
#    → Vercel staging に自動デプロイ
#    → migration があれば Neon staging に自動適用
#    → https://clay-shooting-stg.vercel.app で動作確認

# 8. 確認OKなら main 向けPR作成
gh pr create --base main --head staging --title "Release: ..."

# 9. main にマージ
#    → Vercel 本番に自動デプロイ
#    → migration があれば Neon production に自動適用
#    → https://clay-shooting.vercel.app で反映確認
```

### 旧フロー（v3.78.4 まで、参考）
```bash
git push origin main  # 直接 main に push、即本番デプロイ
```
→ v3.78.5 から `main` / `staging` ブランチ保護により直接 push 不可。

## GitHub Actions（CI/CD）

### `.github/workflows/ci.yml`
- トリガー: PR + push to main/staging
- ジョブ: TypeScript型チェック・ESLint・Next.js ビルド

### `.github/workflows/migrate-staging.yml`
- トリガー: `staging` への push（`db/migrations/**` 変更時のみ）
- 動作: `scripts/run-migrations.js` で Neon staging branch に未適用 migration を自動実行
- 必要な GitHub Secret: `STAGING_DATABASE_URL_UNPOOLED`

### `.github/workflows/migrate-production.yml`
- トリガー: `main` への push（`db/migrations/**` 変更時のみ）
- 動作: 同じスクリプトで Neon production に未適用 migration を自動実行
- 必要な GitHub Secret: `PRODUCTION_DATABASE_URL_UNPOOLED`

### `.github/workflows/keep-alive.yml`
- 既存の Neon DB スリープ防止（毎週水曜 3:00 JST）

## DBセットアップ

### 初回セットアップ
1. Neonダッシュボード → プロジェクト「Clay-shooting」を選択
2. 「SQL Editor」を開く
3. `db/schema.sql` を実行（初期テーブル作成）
4. `db/migrations/001〜023.sql` を順番に実行

### 新規 migration 追加時
1. `db/migrations/NNN_xxx.sql` を作成（連番、ゼロパディング）
2. **Node.js + @neondatabase/serverless で自動実行**
   ```bash
   # 例: scripts/run-migration.js を作成して
   node scripts/run-migration.js db/migrations/024_xxx.sql
   ```
3. ⚠️ **memory feedback (Neon migration)**: ユーザーに依頼せず Claude が自分で実行する
4. 実行後、`docs/05_開発履歴.md` に migration 番号と実行済の旨を明記

### Migration 実行履歴（023まで全実行済み）

| No. | 概要 | 状態 |
|-----|------|------|
| 001〜023 | 詳細は `docs/03_DBスキーマ.md` の一覧参照 | ✅ Neon実行済 |

## デプロイ後の動作確認チェックリスト

git push 後、以下を確認:

- [ ] Vercel ダッシュボードでビルドが「Ready」状態
- [ ] フッターのバージョン番号が `lib/version.ts` と一致
- [ ] `/admin/login` でログイン可能
- [ ] 公開大会一覧が表示される
- [ ] 申込フローでメールが届く（テスト大会で確認）
- [ ] 操作ログが記録されている

> **重要（memory feedback）**: git push 後は必ず Vercel API でデプロイ結果を確認し、エラー時は即修正する。フッターのバージョン番号で反映を確認するのが最終チェック。

## トラブルシューティング

### NextAuth でログインできない
- `NEXTAUTH_SECRET` 設定漏れ → 32文字以上のランダム文字列を Vercel に設定
- `NEXTAUTH_URL` が本番URLと不一致 → 一致させる
- Google OAuth リダイレクトURI が設定されていない → Google Cloud Console で追加

### メール送信失敗
- `EMAIL_PASS` がアカウントパスワードになっている → アプリパスワードに変更
- Gmail送信制限（1日500通）に到達 → 監視

### `/admin/*` にアクセスして即ログイン画面に飛ぶ
- セッション切れ（JWT有効期限） → 正常動作
- middleware.ts の matcher が `/admin/:path*` を含むことを確認
