# トラブルシューティング集（v3.78.8 時点）

3環境運用（local / staging / production）でよく遭遇するエラーと対処法をまとめます。

---

## 🐳 ローカル開発（Docker）

### `docker compose up` で `.env.local` が見つからないエラー
```
Error response from daemon: failed to load env file: open .env.local: no such file or directory
```
**対処**: `cp .env.example .env.local` してから値を編集。

### Next.js が起動するが http://localhost:3000 にアクセスできない
**原因**: コンテナ内 Next.js が `localhost` にバインドしている。
**対処**: `package.json` の `dev:docker` に `--hostname 0.0.0.0` が含まれていることを確認。

### ファイル変更してもホットリロードしない
**原因**: Windows Docker Desktop でのファイル監視は polling が必要。
**対処**: `Dockerfile` の dev ステージに以下があることを確認:
```dockerfile
ENV WATCHPACK_POLLING=true
ENV CHOKIDAR_USEPOLLING=true
```

### `node_modules` が壊れた / 依存追加後に動かない
```bash
docker compose down
docker compose build --no-cache
docker compose up
```

### Neon dev branch に接続できない
**原因**: `DATABASE_URL` が本番のものになっている、または expired。
**対処**:
1. https://console.neon.tech → Branches → `dev-local` の接続文字列を再コピー
2. `.env.local` の `DATABASE_URL` と `DATABASE_URL_UNPOOLED` を更新
3. `docker compose restart`

## 🌐 GitHub Actions / CI

### `npm ci can only install packages when your package.json and package-lock.json are in sync`
**原因**: ローカルで `npm install --ignore-scripts` 等で部分インストールするとロックファイルが不整合になる。
**対処（ローカルで）**:
```bash
rm -rf node_modules package-lock.json
npm install
git add package-lock.json
git commit -m "fix: regenerate lock file"
git push
```

### TypeScript 型チェックジョブが失敗
**対処**:
1. ローカルで `npx tsc --noEmit --incremental false` を実行
2. エラーが出たファイルを修正（特に `lib/types.ts` 変更時の波及に注意）
3. 再 push

### ESLint ジョブが失敗（warningでない error）
**対処**:
1. ローカルで `npx eslint .` を実行
2. エラー（赤）のみ修正、警告（黄）は無視可
3. もし新ルールが厳しすぎる場合は `eslint.config.mjs` で `'rule-name': 'warn'` に降格

### Next.js ビルドが「環境変数が無い」で失敗
**原因**: `next build` 時に `NEXTAUTH_SECRET` 等が未設定。
**対処**: `.github/workflows/ci.yml` の `build` ジョブ `env:` にダミー値が入っていることを確認。

### `actions/checkout` や `actions/setup-node` が deprecated 警告
**対処**: Node.js 24 への移行が予定されている（2026-09）。それまでに `@v5` 等にアップグレード。

## 🗄️ Neon / Migration

### Migration 自動実行ジョブが失敗（`relation "_migrations" does not exist`）
**原因**: 初回セットアップで migration 024 がまだ適用されていない。
**対処**:
1. Neon SQL Editor で `db/migrations/024_create_migration_history.sql` を手動実行
2. その後の migrate ジョブが自動的に動作する

### Migration 自動実行ジョブが失敗（`Missing: STAGING_DATABASE_URL_UNPOOLED`）
**対処**:
https://github.com/matsuo-mitsuhiro/clay-shooting/settings/secrets/actions
で以下を登録:
- `STAGING_DATABASE_URL_UNPOOLED` — Neon staging branch の **プール無し** 接続文字列
- `PRODUCTION_DATABASE_URL_UNPOOLED` — 本番の **プール無し** 接続文字列

### Neon が "compute paused" になる
**原因**: 一定期間アクセスが無いと Neon が自動的に compute を停止する。
**対処**:
- 本番は `.github/workflows/keep-alive.yml`（毎週水曜 3:00 JST）で自動 ping
- staging も必要なら同様の workflow を追加

### Migration が途中で失敗してロールバックされた
**原因**: SQL に文法エラー、または既存データに違反する CHECK / UNIQUE 制約。
**対処**:
1. ジョブログで失敗箇所を確認
2. SQL ファイルを修正
3. 通常の migration として再実行（`scripts/run-migrations.js` がトランザクション保護しているので安全）

## 🚀 Vercel

### Staging へのデプロイが失敗（環境変数不足）
**対処**: Vercel ダッシュボード → `clay-shooting-stg` プロジェクト → Settings → Environment Variables で全8変数（`DATABASE_URL` / `NEXTAUTH_*` / `GOOGLE_*` / `SYSTEM_ADMIN_EMAILS` / `EMAIL_*`）を確認。

### `clay-shooting-stg.vercel.app` で Google ログインが `redirect_uri_mismatch`
**対処**:
1. https://console.cloud.google.com/apis/credentials
2. 既存の OAuth 2.0 クライアント ID をクリック
3. 「承認済みのリダイレクト URI」に追加:
   - `https://clay-shooting-stg.vercel.app/api/auth/callback/google`
   - `http://localhost:3000/api/auth/callback/google`（ローカル用）

### Vercel の「Production Branch」設定ミスで本番 URL が staging を指す
**対処**: 各プロジェクトの Settings → Git で:
- 本番プロジェクト（`clay-shooting`）: Production Branch = `main`
- staging プロジェクト（`clay-shooting-stg`）: Production Branch = `staging`

### Vercel preview deployments が増えすぎてストレージ圧迫
**対処**: Vercel ダッシュボード → Settings → Git → 「Limit deployments」で古いものは自動削除設定可。

## 📧 メール送信

### staging から本物のメールが届いてしまう（事故）
**原因**: `EMAIL_USER` / `EMAIL_PASS` が Gmail（本番）のままになっている。
**対処**: staging Vercel の環境変数を **Mailtrap.io の SMTP** に差し替える:
```
EMAIL_USER=（Mailtrap inbox の Username）
EMAIL_PASS=（Mailtrap inbox の Password）
```
さらに `lib/email.ts` の SMTP host が Mailtrap を指す必要があるかもしれない。

### Gmail の送信制限（1日500通）に到達
**対処**: 重要時のみ動作する設計に変更、または SendGrid 等の専用サービスへ移行。

## 🧪 Playwright E2E

### スモークテストが timeout で失敗
**原因**: ローカル Next.js dev サーバー起動が遅い、または `webServer.timeout` が短い。
**対処**: `playwright.config.ts` の `webServer.timeout` を 120_000 → 180_000 に増やす。

### `getByLabel(...)` で要素が見つからない
**原因**: `<label>` が `htmlFor` 属性で `<input>` と紐付いていない（v3.78 時点で本プロジェクトの全フォームで未対応）。
**対処**: `getByPlaceholder('...')` または `page.locator('input[type="..."]')` を使う。

### Mailtrap からテストメールを取得する方法（フローテスト用）
```typescript
const res = await fetch(
  `https://mailtrap.io/api/accounts/<account_id>/inboxes/${MAILTRAP_INBOX}/messages`,
  { headers: { 'Api-Token': MAILTRAP_TOKEN } }
);
const messages = await res.json();
const latest = messages[0]; // newest first
const code = latest.html_body.match(/(\d{6})/)?.[1];
```

## 🔀 Git / PR ワークフロー

### 「main に直接 push できない」エラー
```
remote: error: GH006: Protected branch update failed for refs/heads/main.
remote: error: Cannot push directly to a protected branch.
```
**対処**: ブランチ保護ルールが効いている（v3.78.5〜）。`feature/*` ブランチで作業して PR 経由でマージすること:
```bash
git checkout -b feature/<目的>
# ... 作業 ...
git push -u origin feature/<目的>
gh pr create --base staging
```

### PR の base ブランチを変えた後、CI が走らない
**対処**: PR を一度 close → reopen すると workflow が再 trigger される。
```bash
gh pr close <番号> && gh pr reopen <番号>
```

### feature ブランチで複数 PR を積むときの version 衝突
**原因**: 各 feature ブランチが main 起点で `lib/version.ts` を独自に bump すると、merge 時にコンフリクト。
**対処**:
- 後から作る PR は前 PR のブランチを base にする（チェイン）
- もしくは衝突発生時に手動で staging の最新値 +1 に更新
